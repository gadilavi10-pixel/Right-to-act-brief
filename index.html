Protocol Brief · Right-to-Act · v1.0

Not Every AI Action Is Allowed to Exist

A manipulated model output is no longer enough. The attacker must make the entire action space structurally coherent.

Most AI security tools try to detect malicious behavior. Right-to-Act changes the control point: before an AI-driven output becomes an action, the action must prove that it is allowed to exist.

Structural Security Model
Closure Proof Before Execution
Decision Boundary Protocol
System Integrity Aware

Key Terms

Closure Proof means a complete set of conditions required for an action to be allowed.

Legitimacy Envelope means the full context required to evaluate whether a proposed action is structurally admissible.

Action Lattice means the predefined structure of allowed actions, fallbacks, suspensions, and transitions.

Snapshot means the auditable proof of the action’s legitimacy state at the moment of evaluation.

The Problem

The current security model around AI is still mostly reactive. It detects, filters, logs, audits, or blocks after a signal appears suspicious.

But autonomous systems do not only produce information. They increasingly produce proposed actions: payments, account changes, workflow changes, access changes, routing decisions, infrastructure operations, and compliance-sensitive outcomes.

The core question is no longer only: “Is this output safe?”
The deeper question is: “Does this proposed action have the right to become real?”

The Shift

Existing Paradigm

Trust behavior, then monitor

Existing systems often assume that if the model output looks safe enough, the action can proceed, while monitoring and audit happen around it.

Right-to-Act

Prove legitimacy before execution

A proposed action does not proceed because the AI is confident. It proceeds only if a full Closure Proof can be produced against predefined structural conditions.

The protocol does not improve decisions. It defines when a decision is allowed to become an action.

Where It Sits

Right-to-Act is not a model, not a guardrail, and not a post-event audit layer. It is a decision boundary between proposed intent and execution.

01User / System Intent
02AI Produces Proposed Action
03Right-to-Act Boundary
04Closure Proof / Snapshot

How It Works in Practice

Example: an AI agent attempts to send a payment.

01Proposed Action: Send Payment
02Required Conditions: Authorization, Balance, Scope, Compliance
03Missing Condition: Authorization Context
04No Closure Proof
05Action Does Not Execute
The AI may propose the action. The protocol decides whether the action can become real.

Security Impact Map

The protocol does not claim to erase every cyber risk. It changes which attacks can still translate into legitimate actions.

1 · Structural Elimination

Attacks that no longer produce action

  • Autonomous Agent Chain Exploitation / Uncontrolled Agent Chaining
  • Direct Prompt Injection / Jailbreaking
  • Indirect Prompt Injection
  • Data Exfiltration via LLM
  • Unauthorized Action Execution via AI Agents
  • Replay Attacks / Action Reuse
  • Backdoored or Poisoned Model Outputs
  • TOCTOU / Race Conditions around action execution
  • Privilege Escalation through the AI layer
2 · Structural Resistance

Attacks that become much harder to exploit

  • Single-source Data Poisoning
  • Partial / isolated Oracle Manipulation
  • Real-time Data Manipulation
  • Social Engineering
  • Account Takeover (ATO)
  • DDoS against Re-authorization Paths
  • Reconnaissance / System Probing
  • Coordinated Multi-source Condition Shaping
  • Cross-layer Coherence Manipulation
These attacks do not disappear. They must now create structural coherence across multiple conditions, not merely manipulate a single signal.
3 · Hardened Structural Risk

Risks outside the direct action boundary

  • Implementation Bugs in the core or surrounding code
  • Governance / Meta-Protocol Manipulation
  • Hardware-level Spoofing without attestation
  • Infrastructure-level DoS
  • Misconfiguration of Action Lattice or Conditions
With System Integrity controls such as signatures, attestation, separation of authority, and immutable audit, these risks become high-cost, low-probability, and nearly impractical to exploit in normal operation.
4 · Adaptive Calibration

Learning from outcomes without self-modification

  • Continuous condition refinement based on outcomes
  • Reduction of recurring misclassification patterns
  • Improvement of threshold calibration and lattice boundaries
  • Detection of structural inconsistencies over time
  • Increased resistance to multi-source manipulation patterns
The system improves over time by refining conditions and boundaries through governed calibration, not by letting the AI rewrite the protocol at runtime.

Why This Matters

Most attacks against AI systems are dangerous because model outputs can be converted into real-world effects. Right-to-Act breaks that assumption.

Even if the AI is misled, manipulated, overconfident, or operating in a distorted context, the proposed action must still satisfy the Legitimacy Envelope and generate a Closure Proof before execution.

Manipulating the model is no longer sufficient. The action still has to satisfy the full structural conditions.

System Integrity Layer

Right-to-Act proves the legitimacy of an action. A production-grade deployment must also prove the integrity of the system that evaluates it.

Proof of Action

Closure Proof, Snapshot, Legitimacy Envelope, Action Lattice, and conditional continuity.

Proof of System

Signed code and policies, remote attestation, immutable audit, separation of authority, versioning, and controlled governance.

Learning reduces error. Control prevents corruption.

Bottom Line

Right-to-Act is a structural security model for AI-driven action. It does not try to guess the attacker’s intent. It does not rely on model confidence. It does not wait for a post-event audit.

It requires every action to prove its legitimacy before it can become real.

From “detect the attack” to “prevent illegitimate action from existing.”
